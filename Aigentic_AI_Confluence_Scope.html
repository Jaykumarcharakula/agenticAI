01

Executive Summary

📌 Problem in one line

The D&A DLC mailbox receives 20,000+ alerts/month across 20+ categories with no severity tagging, no environment segregation, and no auto-remediation — requiring engineers to manually review every alert.

This document defines a 4-phase Aigentic AI implementation that will automatically segregate alerts by environment (Dev / Test / AppDev / AppTest / Prod), classify them by severity (Critical / High / Medium / Low), auto-remediate safe Critical alerts with zero production impact, and auto-close resolved incidents.

The expected outcome is a ~70% reduction in manual alert handling, a sub-15-minute Critical alert response time, and significant reduction in security risk and engineer on-call fatigue.

⚠️ Key Gaps

No severity classification — every alert looks the same regardless of risk.
No environment segregation — Production and Non-Prod alerts are completely mixed.
No auto-remediation — 100% of alerts require manual engineer review.
Critical alert risk — high-volume noise causes genuine threats to be missed.

03

Objectives

  • Segregate all alerts by environment: Dev · Test · AppDev · AppTest · Prod
  • Classify every alert with a severity level: Critical · High · Medium · Low
  • Auto-remediate Critical alerts that are safe and non-impactful to production
  • Auto-close incidents once resolution is confirmed with a full audit trail
  • Reduce manual engineer toil and Mean Time to Resolve (MTTR)
  • Provide real-time dashboard for alert health, SLA compliance, and model accuracy
04

Scope of Work

In Scope

  • All 20+ alert categories in D&A DLC Notifications mailbox
  • Environment tagging across all 5 tiers: Dev, Test, AppDev, AppTest, Prod
  • Severity classification engine (AI/rule-based hybrid)
  • Auto-remediation playbooks for non-impactful Critical alerts
  • Incident auto-closure with evidence snapshot
  • Real-time observability dashboard and weekly digest

Out of Scope

  • Manual remediation of High / Medium / Low alerts (Phase 1)
  • Changes to upstream AWS alert configurations or SNS topics
  • Third-party ticketing system integrations (future phase)
  • Non-AWS alert sources

Phased Implementation Plan

1

Environment Segregation + Severity Classification

Build ingestion pipeline · Tag by environment · Classify by severity · Route to streams

Weeks 1–6
🎯 Phase 1 Objective

Every alert entering the system will be tagged with (a) the environment it belongs to and (b) its severity level — before any engineer sees it. This is the foundational layer all subsequent phases depend on.

Step 1A — Environment Segregation

Alerts will be classified into one of 5 environment tiers using AWS account ID, resource tags, naming conventions, and SNS topic metadata.

PROD
Live production workloads. Highest priority. Any alert here requires immediate attention or automated safe-fix.
SLA: Immediate response
DEV
Developer sandboxes and feature branches. Lower urgency. Alerts are informational unless Critical security.
SLA: 24–48 hours
TEST
QA and integration test environments. Medium urgency. Blocks pipeline if Critical/High.
SLA: 4–8 hours
APPDEV
Application development tier. Pre-test workloads. Treated similar to Dev with slightly higher priority.
SLA: 12–24 hours
APPTEST
Application testing and UAT. Near-production parity required. Higher urgency than TEST.
SLA: 2–4 hours

Step 1B — Severity Classification

Once environment is known, each alert is classified into one of 4 severity tiers based on alert type, impacted resources, compliance ruleset, and historical pattern analysis.

CRITICAL
15 min
Active threat or data breach risk.
GuardDuty findings · Unauthorized API calls · KMS/S3 perimeter breach · Active Non-Compliance in Prod
HIGH
1 hour
Service degradation or quota breach imminent.
ServiceQuota near-limit · CISBenchmark violation (Prod) · Health report failure
MEDIUM
4 hours
Operational advisory. Action needed but not urgent.
EMR version drift · SQS backlog · Non-Prod compliance gap · Health warnings
LOW
24 hours
Informational. No immediate action needed.
Daily cost reports · Dev/Test S3 perimeter drift · Cactus CA in non-Prod

Intelligent Notification & Escalation

Route classified alerts to the right team via the right channel at the right urgency

Weeks 7–10
  • PagerDuty / Slack / Email routing based on Severity × Environment matrix
  • Critical Prod alerts page on-call engineer immediately; Dev Low alerts go to weekly digest only
  • SLA breach alerting — escalate if Critical alert unacknowledged beyond 15 min
  • Duplicate suppression — deduplicate the same alert firing multiple times within a window
  • Alert grouping — correlate related alerts into a single incident (e.g. 50 S3 perimeter alerts → 1 incident)
