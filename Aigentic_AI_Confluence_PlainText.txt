1. Executive Summary


The D&A DLC Notifications mailbox currently receives 20,000+ alerts per month across 20+ categories — spanning AWS infrastructure, security, compliance, and operational health.

These alerts arrive with no automated severity classification, no environment segregation, and no auto-remediation workflow — requiring engineers to manually review every single alert.

This document defines a 4-phase Aigentic AI implementation that will:

    - Automatically segregate alerts by environment: Dev / Test / AppDev / AppTest / Prod
    - Classify every alert with a severity level: Critical / High / Medium / Low
    - Auto-remediate Critical alerts that are safe and non-impactful to production
    - Auto-close resolved incidents with a full audit trail

Expected outcome: ~70% reduction in manual alert handling, sub-15-minute Critical alert response time, and significant reduction in security risk and engineer on-call fatigue.

2.2 Key Gaps in Current State

    - No severity classification — every alert looks the same regardless of actual risk level
    - No environment segregation — Production and Non-Production alerts are completely mixed
    - No auto-remediation — 100% of alerts require manual engineer investigation
    - No incident auto-closure — resolved issues remain open, inflating noise further
    - Critical alert risk — high-volume noise causes genuine security threats to be missed
    - Engineer toil — thousands of alerts reviewed manually every single day

3. Objectives


    - Automatically segregate all alerts by environment: Dev · Test · AppDev · AppTest · Prod
    - Classify every alert with a severity level: Critical · High · Medium · Low
    - Auto-remediate Critical alerts that are safe and non-impactful to existing production setup
    - Auto-close incidents once resolution is confirmed with a full audit trail
    - Reduce manual engineer toil and Mean Time to Resolve (MTTR)
    - Provide a real-time dashboard for alert health and SLA tracking


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━


4. Scope of Work


4.1 In Scope

    - All 20+ alert categories in D&A DLC Notifications mailbox
    - Environment tagging across all 5 tiers: Dev, Test, AppDev, AppTest, Prod
    - Severity classification engine (AI and rule-based hybrid model)
    - Auto-remediation playbooks for non-impactful Critical alerts
    - Incident auto-closure workflows with evidence snapshots
    - Real-time observability dashboard and weekly digest reporting


4.2 Out of Scope

    - Manual remediation of High / Medium / Low alerts in Phase 1
    - Changes to upstream AWS alert configurations or SNS topics
    - Third-party ticketing system integrations (future phase)
    - Non-AWS alert sources

PHASE 1 — Environment Segregation + Severity Classification
Weeks 1 to 6


Objective:
Every alert entering the system will be tagged with (a) the environment it belongs to
and (b) its severity level — before any engineer sees it. This is the foundational layer
that all subsequent phases depend on.


Step 1A — Environment Segregation

Alerts will be classified into one of 5 environment tiers using AWS account ID,
resource tags, naming conventions, and SNS topic metadata.

Environment     Description                                                              SLA Target
---------------------------------------------------------------------------------------------------
PROD            Live production workloads. Highest priority. Any alert requires          Immediate
                immediate attention or an automated safe-fix.

APPTEST         Application testing and UAT. Near-production parity required.            2–4 hours
                Higher urgency than TEST.

TEST            QA and integration test environments. Medium urgency.                    4–8 hours
                Blocks pipeline if Critical or High.

APPDEV          Application development tier. Pre-test workloads.                        12–24 hours
                Slightly higher priority than Dev.

DEV             Developer sandboxes and feature branches. Lower urgency.                 24–48 hours
                Informational unless Critical security issue.


Environment Detection Logic

Signal Source           Example Value                    Mapped Environment    Confidence
------------------------------------------------------------------------------------------
AWS Account ID          Account registered as prod       PROD                  High
Resource Tag: env       env=production                   PROD                  High
SNS Topic Name          dlc-prod-alerts-*                PROD                  High
Resource Tag: env       env=dev or env=development       DEV                   High
Resource Tag: env       env=test or env=qa               TEST                  High
Resource Tag: env       env=appdev                       APPDEV                High
Resource Tag: env       env=apptest or env=uat           APPTEST               High
Resource Name Pattern   *-nonprod-* or *-dev-*           DEV                   Medium
No Tag or Unknown       —                                PROD (assume worst)   Low — escalate for tagging

IMPORTANT: Any resource that cannot be classified will default to PROD treatment
(assume worst case). A mandatory AWS tagging audit will run in Week 1–2 to identify
and fix all untagged resources.


Step 1B — Severity Classification

Once the environment is identified, each alert is classified into one of 4 severity tiers.

Severity    SLA              Classification Criteria
---------------------------------------------------------------
CRITICAL    15 minutes       Active threat, data exfiltration risk, security boundary
                             breach, active non-compliance in Prod

HIGH        1 hour           Service degradation risk, quota breach imminent,
                             policy violation detected

MEDIUM      4 hours          Operational advisory, version drift, non-urgent
                             compliance gap

LOW         24 hours         Informational, trend monitoring, no immediate
                             action needed

How to read this matrix:
Each cell shows the severity assigned when that alert type fires in that environment.
Example: GuardDuty in PROD = CRITICAL (15 min SLA). Same alert in DEV = MEDIUM (4 hour SLA).

PHASE 2 — Intelligent Notification & Escalation
Weeks 7 to 10


    - PagerDuty / Slack / Email routing based on Severity x Environment matrix
    - Critical Prod alerts page on-call engineer immediately
    - Dev Low alerts go to weekly digest only — no individual notifications
    - SLA breach alerting — escalate if Critical alert unacknowledged beyond 15 minutes
    - Duplicate suppression — deduplicate the same alert firing multiple times in a window
    - Alert grouping — correlate related alerts into a single incident
      (e.g. 50 S3 perimeter alerts consolidated into 1 incident)
